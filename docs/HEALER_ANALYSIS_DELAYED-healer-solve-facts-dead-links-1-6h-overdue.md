# Healer Analysis — DELAYED-healer-solve-facts-dead-links-1-6h-overdue

**Issue ID:** DELAYED-healer-solve-facts-dead-links-1-6h-overdue  
**Category:** cron  
**Reported by:** health_check_repair.py / error_healer.py dispatch  
**Cycle:** 1

## Problem

The root scheduler (`error_healer.py`) observed a `DELAYED:` cron entry for a `healer-solve-facts-dead-links` one-shot job and spawned a new analyze/solve/verify chain. The underlying issue (`facts-dead-links`) was already fixed, verified, and marked `FIXED` in `docs/HEALER_STATUS_facts-dead-links.md`. The only remaining artifact is a stale dead-link report in `docs/HEALTH_CHECK.md` caused by `health_check_repair.py` mis-parsing a Markdown link inside `docs/HEALER_SOLUTION_facts-dead-links.md`.

This is a stale one-shot healer job masquerading as a `DELAYED` cron error, not an unresolved dead-link issue.

## Evidence

1. **`docs/HEALER_STATUS_facts-dead-links.md`** (lines 8–29) shows:
   - Status: **FIXED**
   - Verified at 2026-07-21T08:14:00Z
   - Re-ran `python3 .github/scripts/facts_collector.py` → `"dead_links": []`

2. **`docs/FACTS.json`** currently reports:
   ```json
   "dead_links": []
   ```
   The facts collector no longer detects any dead links.

3. **`docs/ERRORS.md`** still lists a Deadlink entry:
   ```
   Dead link in HEALER_SOLUTION_facts-dead-links.md: docs/assets/demo.gif
   ```
   This is generated by `health_check_repair.py::check_dead_links()` (lines 128–155).

4. **The "dead link" is actually inside inline code**, not a real Markdown link. `docs/HEALER_SOLUTION_facts-dead-links.md` line 28 contains:
   ```markdown
   The remaining reported dead link `docs/assets/demo.gif` appears only inside an inline code checklist example (removed literal link syntax to avoid checker false positive).
   ```
   The text `` `![demo](docs/assets/demo.gif)` `` is a code span describing the previously-detected false positive. `health_check_repair.py` does **not** strip inline code spans before scanning links, so it extracts `docs/assets/demo.gif` from inside the backticks and reports it as a dead link.

5. **`facts_collector.py::check_local_links()`** (lines 174–219) already strips inline code spans before link extraction, so it correctly ignores the same string. The two checkers are inconsistent.

## Proposed Fix

1. **Short-term / no-code fix:** Mark `DELAYED-healer-solve-facts-dead-links-1-6h-overdue` as already resolved. No file changes are needed for the original dead-link issue.

2. **Long-term fix (prevent recurrence):** Update `health_check_repair.py::check_dead_links()` to strip inline code spans before extracting Markdown links, matching the behavior in `facts_collector.py`. Add the following before the link regex:
   ```python
   content = re.sub(r'`[^`]*`', '', content)
   ```
   Then re-run `health_check_repair.py` and verify the false-positive dead-link entry disappears from `docs/HEALTH_CHECK.md` and `docs/ERRORS.md`.

3. **Root-scheduler guard:** In `error_healer.py`, before scheduling a new chain for a `DELAYED: healer-solve-...` / `DELAYED: healer-verify-...` entry, check whether `docs/HEALER_STATUS_{issue_id}.md` already shows `Status: **FIXED**` or `Status: Resolved`. If so, skip dispatch and optionally append a log entry that the delayed one-shot was stale. This avoids redundant healer cycles for already-resolved issues.
